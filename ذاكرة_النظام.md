# ذاكرة النظام لمشروع Khidmati / OnSpace

## تعليمات إلزامية لأي مطور أو وكيل يعمل على المشروع

اقرأ هذا الملف أولًا قبل فحص الكود أو تنفيذ أي تعديل. لا تبدأ بإعادة فحص المشروع كاملًا من الصفر إلا إذا كان المطلوب يخص منطقة غير مغطاة هنا أو تغيّر المشروع بعد آخر تحديث لهذا الملف. استخدم هذا الملف مع `هيكل_المشروع.md` و`سجل_التحديثات.md` كذاكرة تشغيلية مختصرة لتقليل تكرار قراءة كل الصفحات والملفات.

آخر تحديث معروف: 17 أبريل 2026.

## ملخص المشروع

المشروع عبارة عن منظومة خدمات منزلية عربية باسم Khidmati / OnSpace، تتكون من تطبيق Expo Router للمستخدمين ومقدمي الخدمة، ولوحة إدارة Next.js منفصلة داخل `admin-web/`.

الغرض الأساسي:

- العميل يختار خدمة مثل تنظيف أو سباكة أو كهرباء.
- محرك شات عربي يجمع تفاصيل الطلب ويحسب السعر ويختار مزود خدمة.
- يتم إنشاء طلب ومتابعة حالته ومحادثته وتقييمه.
- مزود الخدمة يستقبل الطلبات ويغيّر حالتها ويتابع الأرباح والسحب.
- الإدارة تراقب الطلبات والمستخدمين والخدمات وتدفقات الشات والمحافظ والإعدادات.

## التطبيقات الموجودة

### 1. تطبيق Expo الرئيسي

المسار الجذري هو المشروع نفسه.

تقنيات مهمة:

- Expo SDK 53.
- expo-router للمسارات المبنية على الملفات.
- React Native / React Native Web.
- Context API للحالة العامة.
- NativeWind/Tailwind للأجزاء المدعومة.
- Supabase client ما زال مستخدمًا في تطبيق Expo كطبقة بيانات legacy مع fallback محلي عند غياب متغيرات Supabase.

نقطة الدخول:

- `app/_layout.tsx`: يلف التطبيق بـ Providers التالية بالترتيب: AuthProvider, ThemeProvider, I18nProvider, LocationProvider, NotificationsProvider, MessagesProvider, AppProvider.
- `app/(tabs)/_layout.tsx`: يحدد التبويبات حسب الدور: customer, provider, admin.

### 2. لوحة الإدارة Next.js

المسار: `admin-web/`.

تقنيات مهمة:

- Next.js App Router.
- React 19.
- Tailwind CSS.
- PostgreSQL عبر route server-side.
- Drizzle ORM مستخدم في route إرسال push، بينما بوابة البيانات العامة تستخدم `pg` مع SQL مباشر مضبوط بقائمة بيضاء.

مبدأ الإدارة الحالي:

- الواجهة تستورد `admin-web/src/lib/supabase.ts`، لكنه لم يعد Supabase الحقيقي.
- الملف يحاكي Fluent API الخاص بـ Supabase ويرسل العمليات إلى `/api/data`.
- `/api/data` ينفذ العمليات على PostgreSQL من جهة الخادم فقط.
- أي جدول أو عمود جديد للإدارة يجب إضافته إلى القوائم البيضاء داخل `admin-web/src/app/api/data/route.ts` قبل استخدامه من واجهة الإدارة.

## قاعدة البيانات

قاعدة البيانات الحالية في Replit PostgreSQL عبر `DATABASE_URL`.

ملفات مهمة:

- `server/schema.sql`: السكيمة العملية المتوافقة مع Replit PostgreSQL.
- `scripts/db-push.js`: يقرأ `server/schema.sql` ثم `supabase_seed.sql` ويدفعهما لقاعدة البيانات.
- `supabase_schema.sql`: السكيمة الأصلية المستوردة من Supabase، تحتوي RLS وTriggers وPolicies خاصة بـ Supabase.
- `supabase_seed.sql`: بيانات تأسيسية للخدمات والمناطق وتدفقات الشات والإعدادات.
- `server/db.ts`: Pool و Drizzle instance عامان للسيرفر.

أمر دفع السكيمة:

```bash
npm run db:push
```

لا تغيّر أنواع أعمدة `id` الأساسية بدون فحص قاعدة البيانات الحالية. معظم الجداول تستخدم UUID عبر `gen_random_uuid()` أو مفاتيح نصية مثل `services.id` و`app_settings.key`.

## حالة الهجرة من Supabase إلى Replit PostgreSQL

تم إنجاز التالي:

- تثبيت اعتمادات المشروع ولوحة الإدارة.
- إنشاء Replit PostgreSQL وربطه عبر `DATABASE_URL`.
- إنشاء `server/schema.sql` كسكيمة PostgreSQL قابلة للدفع.
- إنشاء `scripts/db-push.js` وتشغيل `npm run db:push` بنجاح.
- نقل Edge Function الخاصة بإرسال تحديث الطلب إلى Next.js route: `admin-web/src/app/api/send-order-push/route.ts`.
- تحويل لوحة الإدارة لاستخدام API server-side بدل Supabase browser client.
- إزالة اعتماد Supabase الرسمي من `admin-web/package.json`.
- إضافة `latitude` و`longitude` إلى جدول `orders` في سكيمة Replit لأن `ordersService.ts` يستخدمهما.

ما يزال legacy/جزئيًا:

- تطبيق Expo الرئيسي لا يزال يعتمد على `services/supabaseClient.ts` وخدمات كثيرة تستخدم `supabase.from(...)` وRealtime وStorage.
- عند عدم وجود متغيرات Supabase، العميل يستخدم fallback آمن بدل الانهيار، لكن هذا ليس بديلًا كاملًا لكل وظائف الإنتاج.
- Realtime في Expo مبني على Supabase channels؛ إذا تقرر التحول الكامل إلى PostgreSQL فقط، يجب بناء API وWebSocket/SSE أو polling بديل.
- Storage في Expo ما زال مرتبطًا بـ Supabase Storage عبر `services/storageService.ts`.

## متغيرات البيئة المهمة

- `DATABASE_URL`: أساسي ويأتي من Replit PostgreSQL.
- `FUNCTION_SECRET`: سر اختياري لحماية route إرسال push. لا يوضع في client.
- `EXPO_ACCESS_TOKEN`: اختياري لإرسال Expo Push API. لا يوضع في client.
- `EXPO_PUBLIC_SUPABASE_URL` و`EXPO_PUBLIC_SUPABASE_ANON_KEY`: legacy لتطبيق Expo إذا تم تشغيل مسارات Supabase.
- `NEXT_PUBLIC_SUPABASE_URL` و`NEXT_PUBLIC_SUPABASE_ANON_KEY`: لم تعد مطلوبة للوحة الإدارة الحالية بعد استبدال العميل، إلا إذا أعيد تفعيل Supabase الرسمي.

## دورة حياة المستخدم

1. المستخدم يسجل أو يدخل من `contexts/AuthContext.tsx`.
2. إذا Supabase مهيأ، يتم استخدام Supabase Auth وجدول `profiles`.
3. إذا Supabase غير مهيأ، يوجد fallback محلي للتطوير في AsyncStorage/local storage.
4. الدور `role` يحدد الواجهة:
   - `customer`: تبويبات العميل.
   - `provider`: تبويبات مزود الخدمة.
   - `admin`: تبويبات الإدارة داخل Expo ولوحة الإدارة المنفصلة.
5. `app/(tabs)/_layout.tsx` يخفي أو يظهر الشاشات حسب الدور.

## دورة حياة الطلب

1. العميل يبدأ من الشاشة الرئيسية أو الشات ويختار خدمة.
2. `AppContext.startChat()` يحدّث كاش المزودين عبر `providerDirectory` ثم يبدأ `initChatSession()`.
3. `services/chatEngine.ts` يوجه المنطق لموديول الخدمة المناسب داخل `services/chatEngineModules/`.
4. بعد اكتمال بيانات الشات يتم تكوين `orderNumber`, `selectedProvider`, السعر، الموعد، والعنوان.
5. `AppContext.createOrderFromChat()` يستدعي `services/ordersService.ts/createOrder()`.
6. الطلب يبدأ غالبًا بحالة `pending`.
7. مزود الخدمة يغيّر الحالة عبر `updateOrderStatus()`.
8. عند تغيّر الحالة، `AppContext` يدفع إشعارات داخلية وقد يستدعي `sendOrderPush()` لإرسال push خارجي.
9. العميل يرى الطلب في `orders` و`order-detail` ويستطيع التقييم أو طلب استرداد حسب الحالة.

حالات الطلب المهمة الموجودة في المشروع تشمل:

- `pending`
- `accepted`
- `on_way`
- `arrived`
- `in_progress`
- `completed`
- `customer_paid`
- `paid`
- `cancelled`

## دورة حياة المحادثة داخل الطلب

- الملفات الرئيسية: `contexts/MessagesContext.tsx`, `services/orderMessagesService.ts`, `app/order-chat.tsx`, `app/(tabs)/order-chat.tsx`.
- الجدول: `order_messages`.
- يعتمد على Supabase Realtime حاليًا في Expo.
- يتم تتبع unread/delivered/read في الجداول والسياقات.

## دورة حياة الإشعارات

- `contexts/NotificationsContext.tsx`: إشعارات داخلية وسجل إشعارات.
- `services/pushTokensService.ts`: تسجيل Expo push tokens في `push_tokens`.
- `services/pushNotificationsService.ts`: يستدعي API إرسال push.
- `admin-web/src/app/api/send-order-push/route.ts`: يقرأ الطلب والتوكنات من PostgreSQL ويرسل إلى Expo Push API.
- `app/_layout.tsx`: يفتح `order-detail` عند الضغط على إشعار يحتوي `orderId`.

## دورة حياة الموقع والمزودين

- `contexts/LocationContext.tsx`: الحصول على موقع المستخدم، تحديد المنطقة، التعامل مع `regions`.
- `services/providerLocations.ts`: توفر وموقع المزود.
- `services/providerDirectory.ts`, `services/mapProviders.ts`: كاش وقوائم المزودين للخرائط والشات.
- `AppContext` يرسل heartbeat كل 30 ثانية للمزود إذا كان الدور provider ولديه location.

## دورة الحياة المالية

- `services/walletService.ts`: الحسابات والمعاملات وحساب أرباح المزود.
- `services/withdrawService.ts`: طلبات السحب.
- `services/refundService.ts`: طلبات الاسترداد.
- `admin-web/src/app/dashboard/wallet/page.tsx`: إدارة السحوبات والاستردادات يدويًا.
- `app_settings.key = platform_commission_percent`: نسبة العمولة الافتراضية من seed تساوي 15.

## الجداول الأساسية والعلاقات

- `profiles`: المستخدمون والأدوار.
- `provider_locations`: موقع وتوفر المزود. علاقة 1:1 مع `profiles`.
- `orders`: الطلبات. يرتبط بعميل ومزود من `profiles`.
- `order_messages`: رسائل الطلب. يرتبط بـ `orders` وبالمرسل والمستقبل من `profiles`.
- `notifications`: إشعارات محفوظة للمستخدم.
- `push_tokens`: Expo push tokens للمستخدمين.
- `services`: تعريف الخدمات.
- `provider_services`: ربط مزود بخدمة. علاقة many-to-many بين `profiles` و`services`.
- `regions`: مناطق ورسوم إضافية وحدود جغرافية.
- `service_chat_flows`: JSON لتدفقات الشات حسب الخدمة والمنطقة.
- `wallet_accounts`: رصيد المستخدم.
- `wallet_transactions`: حركات المحفظة.
- `withdraw_requests`: طلبات سحب المزودين.
- `refund_requests`: طلبات الاسترداد.
- `app_settings`: إعدادات عامة مثل نسبة العمولة.
- `service_popularity`: إحصاء شعبية الخدمات.
- `user_presence`: حالة الاتصال.

## قواعد تعديل مهمة

- عند تعديل صفحة Expo، افحص الملف المطلوب وسياقه القريب فقط ثم ارجع لهذا الملف بدل فحص كل المشروع.
- عند إضافة صفحة تبويب جديدة، راجع `app/(tabs)/_layout.tsx` لتحديد ظهورها حسب الدور.
- عند إضافة خدمة جديدة، عدّل غالبًا:
  - `constants/config`
  - `services/chatEngine.ts`
  - `services/chatEngineModules/<service>.ts`
  - seed أو لوحة الإدارة `services` و`service_chat_flows`
- عند إضافة جدول للإدارة:
  - أضفه إلى `server/schema.sql`.
  - أضفه إلى whitelist في `admin-web/src/app/api/data/route.ts`.
  - شغّل `npm run db:push`.
- عند إضافة بيانات حساسة لا تستخدم `EXPO_PUBLIC_*` ولا `NEXT_PUBLIC_*` للأسرار.
- عند تعديل server routes أو schema، أعد تشغيل workflow وتحقق من logs.

## نقاط خطر معروفة

- `supabase_schema.sql` يحتوي RLS وSupabase Auth references مثل `auth.users` و`auth.uid()`؛ لا تستخدمه مباشرة كبديل لـ `server/schema.sql` داخل Replit PostgreSQL إلا بعد تنظيفه.
- `admin-web/src/lib/supabase.ts` اسمه Supabase لكنه محاكي محلي فوق `/api/data`، لا تفترض أنه SDK رسمي.
- تطبيق Expo ما زال يضم Supabase Realtime/Storage/Auth legacy؛ لا تعلن أنه مهاجر بالكامل إلى PostgreSQL إلا بعد استبدال هذه الطبقات.
- بوابة `/api/data` قوية لكنها عامة؛ يجب إبقاء whitelist صارمًا وعدم السماح بأسماء جداول أو أعمدة من الطلب مباشرة بدون تحقق.
- لا تنقل أسرار Expo push أو function secret إلى ملفات client.
