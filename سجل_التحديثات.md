# سجل التحديثات والتحليل

هذا الملف يوضح ما تم تنفيذه أثناء تهيئة المشروع على Replit وما يجب أن يقرأه أي مطور قبل بدء فحص جديد.

## تعليمات قراءة

قبل أي تعديل جديد:

1. اقرأ `ذاكرة_النظام.md`.
2. اقرأ `هيكل_المشروع.md`.
3. اقرأ هذا الملف لمعرفة آخر تغييرات البنية والهجرة.
4. افحص فقط الملفات المرتبطة بالتعديل المطلوب، ولا تعيد فحص كل المشروع إلا عند الضرورة.

## تحديثات الهجرة المنفذة

### الاعتمادات والتشغيل

- تم تثبيت اعتمادات الجذر لتطبيق Expo.
- تم تثبيت اعتمادات `admin-web`.
- تم تثبيت `drizzle-orm` و`pg` حيث يلزمان للاتصال بـ PostgreSQL.
- تم تشغيل workflow الخاص بـ Expo بنجاح.
- تم تشغيل workflow الخاص بلوحة الإدارة Next.js بنجاح.

### PostgreSQL و schema

- تم إنشاء قاعدة بيانات PostgreSQL في Replit وربطها بمتغير `DATABASE_URL`.
- تم إنشاء `server/schema.sql` كسكيمة عملية متوافقة مع PostgreSQL بدون الاعتماد على Supabase Auth/RLS.
- تم إنشاء `scripts/db-push.js` لدفع السكيمة والـ seed.
- تم إضافة script في `package.json`:

```json
"db:push": "node ./scripts/db-push.js"
```

- تم تشغيل:

```bash
npm run db:push
```

بنجاح.

- تم لاحقًا إضافة `latitude` و`longitude` إلى جدول `orders` في `server/schema.sql` لأن `services/ordersService.ts` يقرأ ويكتب هذه الحقول.
- تم تحديث whitelist في `admin-web/src/app/api/data/route.ts` للسماح بـ `orders.latitude` و`orders.longitude`.

### لوحة الإدارة

- تم تحويل لوحة الإدارة من Supabase browser client إلى server-side PostgreSQL API.
- `admin-web/src/lib/supabase.ts` أصبح محاكيًا لواجهة Supabase fluent API، ويستدعي `/api/data`.
- تم إنشاء `admin-web/src/app/api/data/route.ts` كبوابة CRUD موحدة.
- البوابة تستخدم whitelist للجداول والأعمدة المسموحة.
- تم إزالة `@supabase/supabase-js` من `admin-web/package.json`.
- صفحات الإدارة الحالية تعمل فوق المحاكي:
  - dashboard overview
  - users
  - providers
  - orders
  - services
  - flows
  - wallet
  - settings

### Push notifications

- تم نقل منطق Supabase Edge Function `send-order-push` إلى:

```text
admin-web/src/app/api/send-order-push/route.ts
```

- route الحالي:
  - يتحقق من `FUNCTION_SECRET` إذا كان موجودًا.
  - يقرأ الطلب من `orders`.
  - يجلب tokens من `push_tokens`.
  - يرسل الرسائل إلى Expo Push API.
  - يستخدم PostgreSQL من جهة الخادم ومعه Drizzle في route.

### ملفات التوثيق والذاكرة

تم إنشاء:

- `ذاكرة_النظام.md`: الذاكرة المرجعية الكاملة للمشروع ودورات الحياة ونقاط الخطر.
- `هيكل_المشروع.md`: الهيكل الكامل للتطبيقات والملفات ومسارات التعديل.
- `سجل_التحديثات.md`: هذا الملف، لتوثيق ما تم تغييره.

## ما لم يكتمل بالكامل بعد

هذه النقاط مهمة حتى لا يفترض أي مطور أن الهجرة انتهت 100%:

- تطبيق Expo لا يزال يستخدم `services/supabaseClient.ts` في معظم خدماته.
- Realtime في Expo ما زال يعتمد على Supabase channels.
- Storage ما زال يعتمد على Supabase Storage.
- Auth في Expo يستخدم Supabase إذا توفرت متغيراته، وإلا fallback محلي للتطوير.
- لوحة الإدارة فقط هي التي تم تحويلها فعليًا إلى PostgreSQL server-side API في هذه المرحلة.

## آخر تحقق تشغيلي

- `Start Expo`: يعمل.
- `Start application`: يعمل.
- `/api/data` تم اختباره بطلب `services` وأرجع بيانات seed بنجاح.
- `npm run db:push` يعمل بنجاح.

## توصيات عند التعديل القادم

### إذا المطلوب تعديل واجهة Expo

ابدأ من:

- `app/(tabs)/...` أو `app/...` حسب الشاشة.
- السياق المرتبط في `contexts/`.
- الخدمة المرتبطة في `services/`.

لا تبدأ من `admin-web` إلا إذا كان التعديل يخص لوحة الإدارة.

### إذا المطلوب تعديل لوحة الإدارة

ابدأ من:

- صفحة dashboard المطلوبة تحت `admin-web/src/app/dashboard/`.
- `admin-web/src/lib/supabase.ts` فقط إذا احتجت دعم method جديد في المحاكي.
- `admin-web/src/app/api/data/route.ts` إذا احتجت جدول/عمود/عملية جديدة.

### إذا المطلوب إضافة جدول أو عمود

1. عدّل `server/schema.sql`.
2. إن كان سيظهر في الإدارة، حدّث whitelist في `/api/data`.
3. شغّل `npm run db:push`.
4. أعد تشغيل workflow إذا كان التعديل يمس server route.
5. حدّث `ذاكرة_النظام.md` إذا كان التعديل معماريًا.

### إذا المطلوب استكمال الهجرة من Supabase

المسار الصحيح:

1. إنشاء API routes لتطبيق Expo أو backend مشترك.
2. استبدال `services/supabaseClient.ts` بخدمات HTTP آمنة.
3. استبدال Realtime بآلية polling أو SSE أو WebSocket.
4. استبدال Storage بآلية رفع ملفات مناسبة.
5. تحديث Auth ليصبح مستقلًا عن Supabase أو ربطه بنظام مصادقة Replit/خادم خاص.
6. بعد كل مرحلة، تحديث `ذاكرة_النظام.md` و`هيكل_المشروع.md`.
